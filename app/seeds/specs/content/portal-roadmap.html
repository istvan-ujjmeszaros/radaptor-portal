Radaptor Roadmap

What is stable, what is being hardened, and what comes next

This roadmap keeps the public demo honest: it separates platform foundations that already work, active hardening work, and the next product-facing capabilities planned for Radaptor.

Stable

Registry-first package flow

Consumer apps run from immutable registry packages by default, while maintainer-only package development stays isolated in local overrides. The split is explicit, testable, and safe for app repositories.

Stable

Widget-composed public and admin pages

The portal homepage, comparison page, request-access flow, login, and admin entry points are built from page specs, layouts, widgets, and explicit resource ACLs instead of ad-hoc controller output.

Hardening now

Migration, export, and cutover safety

The current hardening pass focuses on reliable site snapshots, safer restore flows, worker cutover controls, and explicit guards that prevent content loss during package and application upgrades.

Hardening now

Editable static-site migration workflow

New site implementations start by matching the supplied static design with app-owned layouts, themes, file-backed media, and static widgets before dynamic content models are introduced.

Planned

Media-aware content widgets

The next app-facing layer is a cleaner pattern for galleries, VFS-backed media folders, image presets, and admin-editable content entities that can still render through app-specific templates.

Planned

Visual editing on top of the same model

A drag-and-drop page editor is the natural usability layer, but it should sit on the existing resource, layout, widget, and ACL model instead of bypassing it with a separate builder stack.
