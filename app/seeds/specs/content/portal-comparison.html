Technical Positioning

Why Radaptor feels different from typical MVC flow

Radaptor treats both the use case and the UI contract as first-class objects. One request resolves to one handler, and widgets return explicit tree nodes the CMS can compose today.

Comparison at a glance

The claim is not that Laravel or Symfony cannot be structured cleanly. The claim is that Radaptor makes explicit request handling and composable UI trees the default path.

Concern Typical MVC Framework Flow Radaptor Flow
Request entry point Route resolves to a controller method or callable. URL resolves directly to one concrete Event handler.
Authorization Often distributed across middleware, policies, voters, or helper methods. Part of the Event contract, next to the use case itself.
Use-case boundary Can drift into larger multi-action controllers. Defaults toward one handler per actionable request.
Debugging path Trace routing, middleware, auth layers, and controller conventions. Open the handler and start there.
Architectural drift Easy to accumulate non-local checks and fat entry points. Local request handlers keep behavior easier to inspect.
CMS and UI composition CMS is usually a separate product or an add-on layer on top of the framework. CMS composition is part of the framework model and already builds a component tree.

Built for server-driven UI

Radaptor's CMS is not just a page editor. Each widget owns a simple render contract of template, props, strings, and slots.

HTML is the current renderer, but the composition model is already decoupled from that output. Additional renderers can be added without redesigning how content and layout are authored.

A fair comparison

Laravel and Symfony can both be structured around single-action handlers. Symfony in particular is flexible enough to support a very explicit architecture.

Radaptor's claim is narrower: it starts from that model, so you do not need to impose it against the framework's default gravity.

Why the term “Event”?

In Radaptor, an Event is not a pub/sub event or a DDD domain event.

It is the concrete request handler for one use case. If you prefer ADR-style language, think of it as an action object with an explicit authorization contract.

Practical outcome

Less hidden framework gravity. Less policy scavenger hunt. More direct correspondence between the request and the code that owns it.

The result is a framework that can expose product pages, internal tools, admin screens, and future renderer targets without splitting the mental model in half.
